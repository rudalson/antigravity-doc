# Asynchronous Subagents

서브 에이전트는 복잡한 작업을 병렬화하고 메인 에이전트의 컨텍스트를 보존하는 탁월한 방법입니다. 에이전트가 모든 단계를 순차적으로 실행하는 대신, 테스트 실행이나 광범위한 코드베이스 검색과 같은 작업을 전용 서브 에이전트에 위임할 수 있습니다. 이러한 아키텍처를 통해 부모 에이전트는 다른 작업을 병렬로 계속 진행할 수 있으며, 서브 에이전트의 구체적인 작업 내용으로 인해 부모 에이전트의 컨텍스트 창이 오염되는 것을 방지합니다.

## 서브 에이전트 호출

부모 에이전트는 `invoke_subagent` 도구를 호출하여 전용 역할과 초기 프롬프트를 가진 새 동시 세션을 생성(Spawn)합니다.

* **작업 공간 옵션**: 서브 에이전트는 부모와 동일한 작업 공간을 상속받거나 격리된 Git 작업 트리(worktree)를 생성할 수 있습니다.
* **컨텍스트 격리**: 서브 에이전트는 부모와 동일한 모델을 사용하여 실행되지만, 부모의 기존 대화 기록(컨텍스트 창)을 상속받지 않고 깨끗한 상태에서 시작합니다.
* **실행**: 호출되면 서브 에이전트는 즉시 작업을 실행하기 시작합니다. 부모 에이전트는 언제든지 여러 서브 에이전트를 호출할 수 있습니다.
* **모니터링**: 서브 에이전트 패널을 통해 해당 대화에 접속하여 서브 에이전트의 진행 상황을 직접 모니터링할 수 있습니다.

## 서브 에이전트 라이프사이클 및 상태

서브 에이전트는 백그라운드에서 비동기식으로 실행되므로 부모 에이전트가 작업을 위임하고 즉시 자체 작업을 재개할 수 있습니다. 서브 에이전트는 언제든지 다음 세 가지 상태 중 하나로 존재합니다.

### 1. Running (실행 중)
서브 에이전트가 적극적으로 작업을 실행하고, 도구를 호출하며, 응답을 생성하고 있는 상태입니다.
* **취소**: 서브 에이전트 패널에서 **Stop Subagent (서브 에이전트 중단)** 버튼을 클릭하여 실행 중인 서브 에이전트를 취소할 수 있습니다. 즉시 생성이 취소되고 서브 에이전트는 대기(Idle) 상태로 전환됩니다.
* **부모의 제어**: 부모 에이전트도 서브 에이전트에 메시지를 보내 실행을 중단(Interrupt)시키거나 서브 에이전트를 완전히 종료(Kill)할 수 있습니다.

### 2. Idle (대기)
서브 에이전트가 작업을 완료하고 결과가 포함된 메시지를 부모 에이전트에게 보낸 후 실행을 멈춘 상태입니다.
* **재활성화**: 대기 상태의 에이전트는 다른 에이전트(부모 에이전트가 아니어도 됨)로부터 메시지를 받으면 깨어나 다시 *Running* 상태로 돌아갑니다.
* **컨텍스트 유지**: 다시 깨어날 때, 에이전트는 이전 작업의 모든 컨텍스트를 그대로 유지합니다.

### 3. Killed (종료됨)
서브 에이전트가 영구적으로 종료되었으며 다시 깨울 수 없는 상태입니다.
* **정리**: 서브 에이전트를 위해 생성되었던 임시 Git 작업 트리는 자동으로 정리됩니다.
* **조회**: 사용자 및 다른 에이전트는 종료된 서브 에이전트의 대화 기록 기록(transcript)을 여전히 볼 수 있습니다.

## 에이전트 간 통신

에이전트들은 고유한 에이전트 ID를 사용하여 서로 메시지를 전송함으로써 통신합니다.

* **유연한 라우팅**: 에이전트는 직속 부모나 서브 에이전트뿐만 아니라, ID를 알고 있는 다른 모든 활성 에이전트와도 통신할 수 있습니다.
* **자동 깨움**: 대기 상태의 에이전트가 메시지를 받으면, 새 정보를 처리하기 위해 자동으로 다시 깨어납니다.
* **기록 공유**: 에이전트들은 서로의 대화 기록(transcript)을 볼 수 있어, 협업 워크플로우에 대한 종합적인 시야를 제공합니다.

## 내장 서브 에이전트 vs 커스텀 서브 에이전트

### 내장 서브 에이전트
Antigravity에는 다음과 같은 몇 가지 특화된 내장 서브 에이전트가 기본 제공됩니다.
* **`research`**: 코드베이스 탐색, 내비게이션 및 리서치에 최적화되어 있습니다.
* **`browser`**: 격리된 웹 브라우저를 작동하여 대화형 브라우저 작업을 수행합니다 (`/browser` 슬래시 명령을 통해서만 호출됨).
* **`self`**: 호출한 에이전트의 복사본으로, 동일한 시스템 프롬프트와 도구 세트를 공유합니다.

### 커스텀 서브 에이전트
에이전트는 `define_subagent` 도구를 사용하여 자신만의 커스텀 서브 에이전트를 동적으로 정의할 수 있습니다.
* **설정**: 커스텀 시스템 프롬프트와 함께 읽기 전용, 쓰기(터미널 명령 실행 포함), 서브 에이전트 위임 권한에 대한 특정 도구 세트를 정의합니다.
* **범위**: 한 번 정의되면 해당 대화의 남은 기간 동안 커스텀 서브 에이전트를 반복해서 호출할 수 있습니다.

서브 에이전트는 자신만의 서브 에이전트를 호출할 수 있어, 다중 레이어의 위임 및 계층적 팀 구조를 구성할 수 있습니다.

## 권한 및 설정 상속

서브 에이전트는 부모의 안전 설정을 상속받아 견고한 보안 경계를 유지합니다.

* **상속된 범위**: 서브 에이전트는 부모의 허용된 터미널 명령 접두사 및 파일 읽기/쓰기 디렉토리 범위를 자동으로 상속받습니다. 서브 에이전트는 사용자가 부모에게 이미 승인하지 않은 작업은 수행할 수 없습니다.
* **작업 공간 액세스**: 부모 에이전트는 격리된 Git 작업 트리에서 작동 중인 서브 에이전트의 작업 공간을 포함하여 모든 서브 에이전트의 작업 공간에 대해 전체 액세스 권한을 유지합니다.
* **권한 버블링**: 서브 에이전트가 명시적인 사용자 승인이 필요한 도구 호출을 발견하면, 해당 요청은 사용자의 승인을 받기 위해 서브 에이전트 패널 UI로 자동으로 버블링(전달)됩니다.
